# 开发规范模板

适用范围：新项目初始化时直接复制使用；也可作为现有项目统一开发规范的基础版本。

## 1. 目标
- 保证交付稳定、可回滚、可追踪。
- 降低沟通成本、认知成本、Token 消耗。
- 让新成员能快速按统一方式开发、验证、发布。

## 2. 使用方式
1. 复制本文件到新项目根目录。
2. 按项目技术栈补齐“后端规范”“前端规范”“测试规范”“发布规范”。
3. 删除不适用条目，只保留项目真实会执行的规则。
4. 在 README 中放置规范入口，作为默认开发入口。

## 3. 目录边界
- 后端：`backend/`、`server/`、`api/`、`service/`、`model/`、`db/`
- 前端：`frontend/`、`web/`、`src/`
- 配置：`config/`、`.env*`、`settings*`
- SQL/脚本：`sql/`、`scripts/`
- 文档：`docs/`
- 忽略：`node_modules`、`dist`、`build`、`target`、`__pycache__`、`.venv`

## 4. 开发流程
1. 先定位，再读取，最后修改。
2. 优先最小改动，避免无关重构。
3. 多个同类变更尽量一次完成。
4. 修改后必须做最小验证。
5. 输出必须包含：改了什么、为什么改、怎么验证。

## 5. 低 Token 细则
- 先用 `rg` 定位，再读必要文件。
- 不全盘扫文件，不重复读同一大文件。
- 需要多处信息时并行读取。
- 不做无意义解释，不重复背景。
- 回复尽量短，优先结论和结果。

## 6. 后端规范
- 路由层只做参数校验、鉴权、转发。
- 业务逻辑放到 service/usecase 层。
- 数据访问统一封装，禁止在路由里散落 SQL。
- SQL 必须参数化，禁止字符串拼接用户输入。
- 长耗时任务必须有超时、分页、限流或扫描范围限制。
- 配置项统一进环境变量和示例配置。
- 错误信息要可读，避免吞掉根因。

## 7. 前端规范
- 首屏优先：先核心内容，后次要内容。
- 重组件按需加载，避免全局引入重依赖。
- 接口请求必须有超时和错误兜底。
- 可复用逻辑抽成 utils / composables / hooks。
- 样式只在必要时改，优先保持视觉一致性。
- 文案、标题、行高、对齐属于可用性问题，需认真处理。

## 8. 测试与验证
- 前端至少执行构建验证。
- 后端至少执行语法校验或最小运行验证。
- 涉及逻辑改动时，补充最小回归场景。
- 涉及性能改动时，记录前后对比指标。

## 9. 提交规范
- Commit 格式：`type(scope): summary`
- 常用类型：`feat`、`fix`、`refactor`、`perf`、`docs`、`chore`
- 每次提交应尽量能独立回滚。

## 10. 发布规范
- 默认小步发布。
- 先灰度，再全量。
- 出现问题先回滚可疑改动，避免扩大影响面。

## 11. 新项目初始化清单
- 建立目录边界说明。
- 配置开发流程和低 Token 细则。
- 补齐后端/前端/测试/发布规范。
- 补齐 README 快速启动、环境变量、常用命令。
- 确认构建脚本、启动脚本和验证脚本可运行。
