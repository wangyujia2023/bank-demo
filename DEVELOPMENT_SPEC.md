# 开发规范（强制）

适用范围：本仓库后续所有功能开发、修复、重构、联调与发布。

## 1. 目标
- 保证稳定交付：功能可用、可回滚、可追踪。
- 降低沟通和推理成本：统一代码与提交习惯。
- 降低 Token 消耗：减少无效上下文、重复读取、重复确认。

## 2. 目录边界
- 后端开发范围：`/backend/api`、`/backend/service`、`/backend/doris`、`/backend/middleware`、`/backend/settings.py`、`/backend/app.py`
- 前端开发范围：`/frontend/src`
- SQL 范围：`/sql`、`/backend/sql`
- 忽略目录：`node_modules`、`dist`、`__pycache__`、`log`、`.venv`

## 3. 开发流程（必须）
1. `rg` 定位目标函数/接口/组件后再读文件，禁止全盘扫读。  
2. 一次性批量修改同类问题，减少多轮碎改。  
3. 修改后必须做最小验证：  
   - 前端：`npm run build`  
   - 后端：目标文件语法校验或最小可运行验证  
4. 输出变更清单：改了什么、为何改、如何验证。  

## 4. 后端规范（FastAPI）
- 路由层只做参数校验与转发，业务逻辑放 `service`。
- SQL 必须参数化，禁止字符串拼接用户输入。
- 高耗时查询必须提供约束（时间窗口、limit、白名单事件等）。
- 新增开关一律走 `settings.py + .env.example`，默认值偏安全/低负载。
- 异常要可读：返回业务可理解错误，不吞异常根因。

## 5. 前端规范（Vue3）
- 页面首屏优先：先加载核心数据，次要数据异步补齐。
- 接口请求必须有超时与错误提示；可缓存的数据优先缓存。
- 图表/重组件按需加载；避免全局注入重依赖。
- 仅在 UI 修复时调整样式；功能迭代优先改逻辑与数据流。
- 文案、标题、控件高度等基础可读性问题必须同步修复。

## 6. 低 Token 执行细则（必须）
- 检索优先：先 `rg` 后 `cat/sed`，只读必要片段。
- 禁止重复读同一大文件；二次定位用 `rg -n` 到行号。
- 多文件读取/验证并行执行（可并行就并行）。
- 同类改动一次提交，不拆成多轮问答确认。
- 回复模板固定为三段：  
  1) 结论  
  2) 变更文件  
  3) 验证结果  
- 避免长解释与背景复述；仅保留决策与结果。

## 7. 提交与发布
- Commit 信息格式：`type(scope): summary`  
  - `feat` / `fix` / `refactor` / `perf` / `docs` / `chore`
- 每个提交必须可独立回滚。
- 涉及性能问题的改动，必须附“前后对比指标”（至少 1 个）。

## 8. 性能基线建议
- 前端：关注首屏时间、主包体积、慢页面接口数。
- 后端：关注 P95/P99、慢 SQL 数、超时率。
- 新增功能不得默认扩大扫描范围；优先限定近 N 天。

## 9. 违规处理
- 不满足本规范的改动视为未完成；需补齐验证与规范化后再合入。
